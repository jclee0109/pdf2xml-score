# PDF 악보 → 정확한 MusicXML 변환기 MVP PRD

> **마지막 업데이트:** 2026-04-21 (design doc 반영 — office-hours 세션 기반)

---

## 1. 문서 목적

**PDF/이미지 형태의 악보를 기존 악보 프로그램에서 바로 활용 가능한 clean MusicXML로 변환**하기 위한 MVP의 개발 범위를 정의한다.

초기 목표는 **완전한 악보 편집 툴을 만드는 것**이 아니라, **AI가 80% 변환하고 사람이 나머지 오류를 빠르게 검수해서 실제로 쓸 수 있는 파일을 만드는 레이어**를 만드는 것이다.

---

## 2. 배경 및 문제 정의

### 확인된 수요

- **1차 확인 사용자:** 기획사 CEO (A&R + 작곡 담당)
  - 직접 의뢰 — "이게 있으면 좋겠다"가 아닌 명시적인 build request
  - 보컬 키 전조 + 악보 프로그램 편집이 주요 워크플로우
  - MuseScore 내장 import / 유명 온라인 변환기 (유료/무료 모두) 써봄 → 전부 포기
  - 현재 폴백: PDF 보면서 노트/가사/코드 전부 손 입력, **1회당 30분 ~ 2시간**
- **갭:** 아직 1명만 확인됨. 시장 검증을 위해 **3-5명 추가 인터뷰 필요** (세션 연주자, 강사, CCM 편곡자)

### 실제 사용자 시나리오

기획사 작곡가·편곡가는 다음과 같은 워크플로우를 반복한다.

1. 인터넷에서 악보 PDF를 구한다 (보컬 악보, 피아노+보컬, 리드시트 등)
2. MuseScore / Dorico 등 악보 프로그램에서 활용하려 한다
3. PDF → MusicXML 자동 변환을 시도하지만 결과가 너무 깨져 있다
4. 결국 악보를 보면서 처음부터 손으로 다 입력한다

**자동 변환 결과가 너무 부정확해서, 고치는 시간이 처음부터 입력하는 시간과 같다.**

### 현재 문제

- PDF는 보기에는 편하지만 악보 프로그램에서 편집·활용이 불가능하다
- 기존 PDF → MusicXML 변환 결과에 오류가 너무 많다
  - 코드 심볼이 가사로 들어가거나 아예 사라짐
  - 가사와 텍스트가 뒤섞임
  - 조표 / 키 감지가 틀림
  - 마디 구조가 어긋남
  - 반복 기호 깨짐
- **핵심 실패 패턴:** 도구들이 0%가 아니라 음표는 어느 정도 인식하지만, 코드 심볼과 구조가 틀려서 결국 모든 항목을 검수해야 함. 정확도 ~75-80% 미만 + 오류 표시 없음 → 변환이 수고를 줄이는 게 아니라 오히려 늘림

### 핵심 인사이트

> **PDF를 완벽하게 자동 인식하는 것**보다,  
> **AI가 80% 변환하고, 사람이 나머지 오류를 빠르게 검수해서 실제로 쓸 수 있는 파일을 만드는 것**이 더 현실적이고 가치 있다.

## 3. 경쟁사 분석

### 기존 도구의 한계

| 도구 | 형태 | 가격 | 코드 심볼 | 검수 UI | 주요 문제 |
|------|------|------|----------|--------|---------|
| SmartScore | 데스크탑 | $300+ | **없음** | 제한적 | 수동 수정 50%+, 학습 곡선 |
| PhotoScore | 데스크탑 | $200+ | **없음** | 제한적 | 오류율 높음, 자주 오배치 |
| PlayScore 2 | 모바일 | 프리미엄 | 기타코드만 | 없음 | key 변경 감지 실패 |
| Audiveris | 오픈소스 | 무료 | **없음** | 존재하나 복잡 | 개발자 전용, 핸드라이팅 불가 |
| MuseScore import | 내장 | 무료 | 거의 동작 안 함 | 없음 | 실험적 기능 수준 |
| Melogen AI / PDFtoMusicXML | 온라인 | 무료 | 없음 | **전무** | 변환만 하고 끝 |

### 미확인 경쟁사

- **Newzik LiveScores** — PDF를 인터랙티브 악보로 변환한다고 알려짐. 코드 심볼 처리 여부, 실제 MusicXML export 품질 미확인. **빌드 전 확인 필요.**

### 공통된 실패 패턴

1. **코드 심볼 처리** — 어떤 상용 도구도 제대로 인식하지 못함
2. **검수 경험** — 변환 결과를 빠르게 검수·수정하는 UX가 없거나 매우 불편함

### 시장 공백 요약

> **정확한 변환 결과 + 빠른 검수 UI** = 현재 어떤 도구도 제공하지 않는 조합

## 6. MVP 범위

### 포함 범위

- 리드시트 (단선율 + 코드 심볼)
- 보컬 악보 (멜로디 + 코드 심볼 + 가사)
- 피아노+보컬 2단 편성
- 찬송가 / CCM류 단순 편성
- 비교적 깨끗한 PDF 또는 이미지 입력
- 인식 대상 악보 요소: 음표 (음높이 + 길이), 코드 심볼, 가사, 조표, 박자표, 반복 기호
- MusicXML export (MuseScore 4.x / Dorico 5.x import 기준)

### MVP 제외

- 오케스트라 풀스코어 / 복잡한 피아노 솔로 / 3성부 이상 복잡 편성 → Phase 2
- **전조 기능** — MusicXML export 후 MuseScore/Dorico에서 처리. 제품이 직접 제공하지 않음
- 다이나믹스, 아티큘레이션, 핑거링, 복잡한 꾸밈음
- 특수 현대악보 기호
- 완전한 notation editor
- DAW 수준 재생 / 편집 기능

### 확장 설계 원칙

- 악보 복잡도(성부 수, 악기 수)에 따라 검수 항목이 동적으로 달라지는 구조
- 내부 데이터 모델이 풀스코어의 다성부 구조를 처리할 수 있도록 확장 가능하게 설계

## 7. 채택 아키텍처

### Approach A — 기존 OMR + Claude 멀티모달 코드 후처리 + 검수 레이어 (확정)

기존 OMR 엔진(Audiveris, oemer 또는 상용 API)을 구조 인식에 사용하고, Claude 멀티모달을 코드 심볼 전용 후처리 레이어로 적용한다. 그 위에 검수 UI를 얹는다.

```
PDF in
→ OMR 엔진 선택: Audiveris / oemer / 상용 API 중 실제 PDF 3-5개로 벤치마크
  → 구조 정확도 ≥75% 달성 엔진 선택
→ 선택된 OMR → rough MusicXML (음표, 마디, 조표, 박자표)
→ Claude 멀티모달:
  - 원본 PDF 이미지 + OMR 출력 함께 입력
  - 코드 심볼 식별 및 위치/명칭 추출
  - OMR의 코드 오인식 수정 / 누락 코드 보완
  - 신뢰도 < 임계값이면 해당 텍스트 토큰 전체를 "검수 필요"로 플래그
→ 자동 플래그 규칙:
  - 텍스트 토큰 분류 신뢰도 < 80% (가사 / 코드 / 표현기호 구분 불확실)
  - 감지된 코드들이 조표와 60% 이상 불일치
  - 음표 수가 박자표 기준과 맞지 않는 마디
→ 검수 UI: 플래그된 항목 목록 (전체 스코어 에디터 아님)
  - 코드 목록: 인식된 코드 텍스트 → 사용자 확인 또는 수정
  - 조표: 감지된 key → 사용자 확인 또는 override
  - 텍스트 분류: 모호한 토큰 → 사용자 분류
→ 최종 MusicXML을 MusicXML 4.0 스키마로 검증 후 export
→ 사용자 다운로드 → MuseScore/Dorico에서 바로 활용
```

### 정확도 기준

| 항목 | 기준 | 정답 정의 |
|------|------|---------|
| 구조 (음표/마디) | ≥75% 마디 정확 | 음표 수, 길이 일치 |
| 코드 심볼 | ≥85% (Claude 후처리 후) | pitch class + quality 둘 다 일치 (Cm7 ≠ C7) |

### 플래그 임계값

- 플래그 항목 > 40개이면 검수 시작 전 사용자에게 경고 표시

### 대안 비교

| | Approach A (채택) | Approach B (풀 커스텀 AI) |
|---|---|---|
| 노력 | M — 인간 ~2개월 / CC ~2-3주 | L — 인간 ~4-6개월 |
| 리스크 | 중 — OMR 구조 인식 품질에 의존 | 높음 — 인식 품질 불확실 |
| 장점 | 2-3주 내 CEO에게 테스트 가능 | 구조 인식도 제어 가능 |
| 단점 | 서드파티 OMR 품질에 의존 | 코드 인식 품질 확인까지 오래 걸림 |

## 8. MVP 가치 제안

### 기존 방식

PDF 업로드  
→ 범용 OMR 변환 (코드 심볼 누락, 오류 다수)  
→ MuseScore에서 깨진 결과 확인  
→ 처음부터 손으로 다시 입력 (30분 ~ 2시간)

### 우리 방식

PDF 업로드  
→ AI 1차 변환 (구조 ≥75% + 코드 ≥85%)  
→ 오류 항목만 빠르게 검수 (코드 심볼 / 조표 / 가사 / 마디)  
→ clean MusicXML export  
→ MuseScore / Dorico에서 바로 활용 (전조·편집 포함)

### 핵심 가치

- **"변환만 해주는 도구"가 아닌 "쓸 수 있는 파일을 만들어주는 도구"**
- 완벽 자동화보다 **수정 시간 단축** — 1시간짜리 작업을 10분으로
- 범용 편집기 대체가 아니라 **기존 툴 앞단의 품질 개선**
- **코드 심볼까지 처리하는 유일한 상용 레이어**

## 9. 핵심 기능

### 9.1 입력

- PDF / PNG / JPG 업로드
- 단일 파일 기준 처리
- 파일 크기 제한 설정, 업로드 성공/실패 처리

### 9.2 AI 1차 변환 파이프라인

- OMR 엔진 → rough MusicXML (음표, 마디, 조표, 박자표)
- Claude 멀티모달 → 코드 심볼 인식 및 후처리
- 오류 가능성 높은 항목 자동 플래그 부여
- MusicXML 4.0 스키마 검증

### 9.3 검수 UI ★ (핵심 차별점)

#### 설계 방향

- **전체 스코어 에디터가 아닌 "플래그된 항목 중심 검수 UI"**
- 원본 PDF를 참조용으로 나란히 제공
- 목표: A4 1페이지, 플래그 ≤40개 기준 **검수 완료 5분 이내**
- 플래그 > 40개이면 시작 전 경고

#### 검수 항목 우선순위

**A. 코드 심볼 검수 (1순위)**
- 인식된 코드 심볼 목록 표시
- chord / lyric / text 후보 구분
- 오인식 수정
- 코드를 구조화된 형태로 저장 (전조 / 편곡 가능하도록)

**B. 조표 / key 검수 (2순위)**
- 감지된 key 표시
- 사용자가 key 수정 가능

**C. 가사 / 텍스트 분리 (3순위)**
- lyric / chord / section mark 분류 확인
- 잘못 분류된 텍스트 수정

**D. 마디 구조 검수 (4순위)**
- 마디 수 표시 및 누락 / 병합 후보 알림
- 반복 기호 검수

### 9.4 Export

- MusicXML 다운로드
- MuseScore 4.x / Dorico 5.x import 전제
- export 전 MusicXML 4.0 스키마 검증

- 무료 도구(Melogen AI 등) 대비 차별점: 코드 심볼 정제 품질 + 검수 UX

### B2B 가능성

- 기획사 / 출판사 / 음악 교육기관 팀 플랜
- 대량 악보 변환 API

### 배포

- 웹 앱 (PDF 업로드 → 브라우저에서 검수 → MusicXML 다운로드)
- 모바일 MVP 불필요
- CI/CD: Vercel / Railway, main 브랜치 merge on deploy

## 16. 스프린트 플랜

> **전제: Week 1 스파이크 통과 시**

### Week 1 — 스파이크 + 사용자 검증 (빌드 시작 전)

**기술 스파이크 (2-3일):**
1. CEO에게 직접 손 입력한 악보 PDF 3개 요청 (정답 데이터 포함)
2. 각 PDF를 Audiveris + pdftomusicxml.com에 통과
3. 그 중 하나의 결과 + 원본 PDF 이미지를 Claude에게: *"이 악보 이미지에서 코드 심볼을 모두 찾고, 각 beat 위치와 코드명을 알려줘. OMR 결과의 코드를 수정해줘."*
4. CEO의 정답 데이터와 비교해서 코드 정확도 측정
5. **Go/No-go:** 70%+ → Approach A 계속. 미만 → Approach B 검토

**사용자 인터뷰 (3-4일):**
- 세션 연주자 / 실용음악 강사 또는 학생 / CCM 편곡자 각 1명씩
- 질문: "PDF 악보를 편집하거나 전조해야 할 때 어떻게 하세요?" — 제품 피칭 금지
- **목표:** 3/5 이상이 동일한 pain (손 재입력) 묘사 → 수요 검증

### Week 2 — 코어 파이프라인 구축 (스파이크 통과 시)

- 프론트엔드 스캐폴드: 업로드 폼, 검수 UI 프로토타입 (코드 목록, 키 확인, 텍스트 분류)
- OMR 통합: 선택된 엔진 API 래퍼
- Claude 코드 레이어: 프롬프트 + 구조화된 추출 + MusicXML 패치
- MusicXML 4.0 스키마 자동 검증

### Week 3 — CEO와 실전 테스트

- CEO가 직접 가져온 실제 악보 1개로 end-to-end 테스트
- 검수 UI 반복 개선 (막히는 부분 관찰)
- 스테이징 배포
- CEO 외 독립적인 사용자 1명 추가 테스트

## 19. 최종 포지셔닝

이 제품은 **악보 편집기를 대체하는 제품이 아니다.**  
이 제품은 **"깨진 MusicXML"을 "쓸 수 있는 MusicXML"로 만드는 정제 레이어**다.

- MuseScore / Dorico / Sibelius와 경쟁하지 않고, 그 앞단의 병목을 해결
- 코드 심볼 처리에서 현재 유일한 상용 솔루션 포지션
- 장기적으로는 기존 OMR 엔진 출력 위에 얹히는 후처리 레이어로 확장 가능
