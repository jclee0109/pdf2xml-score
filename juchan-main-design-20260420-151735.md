## Problem Statement

Composers and arrangers in the Korean music industry receive PDF sheet music they need to edit in notation software (MuseScore, Dorico, Sibelius). Existing PDF-to-MusicXML converters have accuracy low enough that reviewing and fixing the output takes as long as re-entering the score from scratch. One CEO/A&R/composer confirmed: 30 minutes to 2 hours per score, tried both paid and free tools, gave up on all of them.

No existing commercial tool handles chord symbols reliably. Chord symbols appearing as lyrics, or disappearing entirely, means the converted file is unusable for transposition or chord-based editing.

## Status Quo

PDF in → existing OMR tool → broken MusicXML (chord symbols missing or wrong, key detection errors, lyrics/text mixed up) → review takes longer than starting from scratch → manual re-entry from PDF anyway.

The failure mode is specific: it's not that tools are 0% accurate. It's that they're accurate enough on notes but wrong enough on chord symbols and structure that you can't trust any part of the output without verifying everything. Below a ~75-80% accuracy threshold with no error surfacing, the conversion creates MORE work, not less.

## Target User & Narrowest Wedge

**Target user:** CEO/arranger at a small entertainment company handling 기획사 A&R work. Needs full-score MusicXML output (piano + vocal, all staves) — not just a lead sheet, because she continues editing in Dorico/MuseScore.

**Wedge:** Full PDF score → MusicXML with a targeted review layer that surfaces the most common errors (chord symbols, key signature, lyric/text separation) without requiring review of every note.

---

## Constraints

- MVP scope: single-staff lead sheets and 2-staff piano+vocal arrangements only. SATB, orchestral, percussion, and complex multi-staff scores are Phase 2.
- Transposition is NOT in the MVP — the product converts to editable MusicXML; transposition happens in MuseScore/Dorico after import.
- In-scope musical elements: notes (pitch + duration), chord symbols above staves, lyrics, key and time signatures, repeat barlines. Out of scope for MVP: dynamics, articulations, fingerings, complex ornaments.
- Output must open and display correctly in MuseScore 4.x and Dorico 5.x without import errors.
- Review step goal: under 5 minutes for an A4-page score with ≤40 flagged items. If flags exceed 40, show the user a warning before they start reviewing.
- No custom OMR training required at MVP — use existing engines + AI post-processing.
- No full notation editor — the review UI surfaces flagged items only, not the full score.

---

## Premises

1. The core workflow problem is real: existing OMR tools produce output that takes as long to review and fix as manual re-entry, making them net-negative vs. typing from scratch.
2. Chord symbols are the most underserved technical gap — no existing commercial tool handles them reliably.
3. The transposition-for-vocalist use case is the sharpest initial trigger — time-pressured, frequent, clearly broken.
4. The right architecture is AI-first + human review layer, NOT trying to achieve 100% automation before shipping.
5. Recognition accuracy must reach ~75-80% before the review layer becomes net-positive. At that threshold, the review UX is the moat — because it determines whether 80% accuracy feels like "almost done" or "still a mess."

## Approaches Considered

### Approach A: Wrap Existing OMR + AI Chord Enhancement + Review Layer (CHOSEN)
Use existing OMR engine (Audiveris, oemer, or a commercial API) for structural recognition. Apply multimodal AI (Claude) as a post-processing layer specifically targeting chord symbols — re-examine the original PDF image and correct the OMR's chord output. Build review UI on top.

**Effort:** M (human: ~2 months / CC: ~2-3 weeks)
**Risk:** Medium — depends on existing OMR hitting the 75-80% baseline on structure

**Pros:**
- Ship fastest — test with the CEO in 2-3 weeks
- AI post-processing for chords is tractable with today's multimodal models
- No need to train a custom OMR model for MVP
- Review UI can be validated with a rough prototype first

**Cons:**
- Dependent on third-party OMR quality for the structural layer
- Chord symbol accuracy still uncertain until tested end-to-end

**Reuses:** Audiveris (open source), oemer (open source), Claude multimodal API for chord correction, any standard MusicXML library

### Approach B: Full Custom AI Pipeline
Build own extraction pipeline using multimodal AI end-to-end. More control, longer path.

**Effort:** L (human: ~4-6 months / CC: ~1 month)
**Risk:** High — recognition quality uncertain before extensive testing

### Approach C: Transposition-Only Wedge
Solve exactly the vocalist transposition use case. Too narrow — user needs full-score output.

## Recommended Approach

**Approach A: Hybrid OMR + AI chord enhancement + review layer.**

Architecture:
```
PDF in
→ OMR engine selection: test Audiveris, oemer, pdftomusicxml.com API on 3-5 
  real PDFs — pick the one that hits ≥75% on note/structure accuracy
→ Selected OMR engine → rough MusicXML (structure: notes, measures, key, time)
→ Claude multimodal: given original PDF image + OMR output, identify chord 
  symbols and correct or add them in the MusicXML
  - Fallback: if confidence < threshold, mark ALL text tokens as "needs review" 
    rather than silently producing wrong chords
→ Auto-flag using these rules:
  - Any text token not classified with >80% confidence as lyric, chord, or expression
  - Key signature if detected chords don't agree with it in ≥60% of measures
  - Measures where note count doesn't match expected time signature
→ Review UI: list of flagged items (not a full score editor)
  - Chord list: show recognized chord text → user confirms or corrects
  - Key signature: show detected key → user confirms or overrides
  - Text classification: show ambiguous tokens → user classifies each
→ Validation: parse final MusicXML against MusicXML 4.0 schema before export
→ Export: corrected MusicXML → user downloads
→ Open in MuseScore/Dorico — ready to use
```

**Accuracy definitions:**
- Structure accuracy (notes, measures): ≥75% of measures correct before review
- Chord symbol accuracy: ≥85% of chords correct after Claude post-processing
- A chord is "correct" if pitch class and quality match ground truth (e.g., Cm7 ≠ C7)

**Spike validation (before building the full product):** Run 3 PDFs through Audiveris + Claude chord correction. Score against known-correct output (the CEO can provide ground truth for PDFs she's previously entered manually). If chord accuracy is below 70%, the approach needs revision before building the review UI.
